\section{Abstract}

Knowledge Base Question Answering (KBQA) allows users to query a structured knowledge base using natural language instead of manually writing SPARQL. This project extends the ChatKBQA generate-then-retrieve framework into a temporal KBQA system that can process questions containing temporal constraints such as \textit{before}, \textit{after}, \textit{during}, \textit{first}, and \textit{last}. The proposed system combines temporal logical-form generation, entity and relation retrieval, knowledge-base execution, and an agentic routing layer that decides whether a question should use the temporal or non-temporal path.

The project is designed as an end-to-end NLP system rather than a notebook-only prototype. It includes a structured data pipeline, modular Python code, training and inference scripts, a FastAPI deployment path, a command-line interface, Docker packaging, and targeted tests. The report documents the business problem, system design, data management decisions, model adaptation strategy, deployment considerations, ethical risks, and project management plan. The comparison, ablation, and deployment sections below are aligned with the implemented repository and the locked run artifacts used for the final submission.

\noindent\textbf{Headline result.} The submitted v1 system completed end-to-end training and evaluation on TempQuestions. Its strongest verified diagnostic result is an \textbf{87.6\% valid S-expression rate} under the broader execution-oriented criterion (995/1,340 valid by the stricter parser criterion is 74.3\%). The raw answer-level run records \textbf{F1 = 0.0}, \textbf{Hits@1 = 0.0}, and \textbf{Accuracy = 0.0}, but this should not be interpreted as a pure generation failure. The v1 system is \textbf{submission-ready as an engineering artifact}, but \textbf{not performance-ready as a production QA model}. The main finding of the submission is that the generation component works substantially better than the end-to-end metric suggests, while the pipeline breaks at the \textbf{grounding/execution stage}: hallucinated relation names prevent most generated S-expressions from being executed against the real knowledge base.

\section{Introduction and Problem Definition}

\subsection{Business Context and Motivation}

Many real users need factual answers that are constrained by time. Examples include analysts asking who held a role before a given person, compliance teams reconstructing timelines, journalists checking event order, and researchers looking for the first or most recent occurrence of a relation. Standard search systems are weak at these tasks because they do not naturally bind temporal expressions to knowledge-base operators. Freebase contains rich entity and relation structure, but the representation is difficult for non-experts to query directly.

This project targets that gap by building a temporal KBQA system over Freebase. The baseline repository, ChatKBQA, already supports a generate-then-retrieve pipeline for non-temporal KBQA. The project contribution is to extend that pipeline so temporal expressions can influence logical-form generation, routing behavior, and answer retrieval without replacing the existing architecture.

The system output is an answer set, plus supporting provenance such as the generated logical form, temporal signals, SPARQL used, and execution path. Manual querying of Freebase requires familiarity with its schema, identifiers, compound value types, and SPARQL syntax. Temporal questions add another layer of complexity because they require ordinal, comparison, or interval reasoning. NLP is needed to map user language to logical forms that can be executed over the knowledge base while preserving the intended temporal semantics.

\section{System Overview and Architecture}

\subsection{Baseline Architecture}

The project preserves the original ChatKBQA design principle:

\begin{center}
\texttt{Question $\rightarrow$ LLM logical-form generation $\rightarrow$ placeholder retrieval $\rightarrow$ resolved S-expression $\rightarrow$ SPARQL $\rightarrow$ answer}
\end{center}

Temporal support is added to this pipeline rather than replacing it with a completely different architecture. This keeps the project aligned with the source system while still contributing meaningful temporal capability.

\subsection{High-Level System Flow}

\begin{figure}[H]
  \centering
  \includegraphics[width=0.92\textwidth]{../figs/F1.drawio.png}
  \caption{High-level system view used in the project documentation}
  \label{fig:system-overview}
\end{figure}

Figure \ref{fig:system-overview} summarizes the end-to-end system flow. At a high level, the system performs the following stages:

\begin{enumerate}[leftmargin=1.4cm]
  \item Detect whether the question contains temporal signals.
  \item Route the question to temporal or standard inference mode.
  \item Generate an S-expression logical form using the fine-tuned LLM.
  \item Retrieve entity and relation bindings for placeholders.
  \item Convert the resolved logical form to SPARQL and execute it.
  \item Return answer and provenance, or retry with relaxed constraints when appropriate.
\end{enumerate}

\subsection{Agentic AI Component}

The agentic component is implemented through the \texttt{TemporalQuestionAgent}. It satisfies the assignment requirement in three ways:

\begin{enumerate}[leftmargin=1.4cm]
  \item \textbf{Multi-step reasoning:} detect temporal signals, route, execute, and refine.
  \item \textbf{Tool usage:} use the KB executor and retrieval stack as external tools.
  \item \textbf{Intermediate decision-making:} retry with relaxed constraints when the first attempt returns no answer.
\end{enumerate}

The routing logic is intentionally simple and auditable. Instead of hiding all behavior inside the model, the system separates question classification, pipeline selection, and fallback behavior at the software layer.

\subsection{Example Agent Interaction}

\begin{quote}
\textbf{Question:} Who was the US president before Obama? \\
\textbf{Step 1:} Detect temporal signal \texttt{before}. \\
\textbf{Step 2:} Route to temporal mode. \\
\textbf{Step 3:} Execute first attempt. \\
\textbf{Step 4:} If empty, retry with relaxed constraints; otherwise return answer with provenance.
\end{quote}

\noindent For the submission, the implemented interaction surfaces are the FastAPI endpoint, the command-line interface, and a lightweight Streamlit demo layer. The API and CLI are the primary inference surfaces; the Streamlit app is a presentation surface that can switch between \textbf{live mode} and \textbf{degraded/offline mode} depending on whether the full model, retrieval assets, and KB backend are available at runtime.

\subsection{Data Pipeline Summary}

The repository documents two important realities:

\begin{enumerate}[leftmargin=1.4cm]
  \item The original human TempQuestions artifacts contain corruption and placeholder-collapse issues.
  \item The practical training pipeline therefore shifted toward synthetic temporal supervision generated from alternative Freebase-aligned sources and filtered before export.

The current working pipeline can be summarized as:

\begin{center}
\texttt{fact source $\rightarrow$ temporal fact normalization $\rightarrow$ template generation $\rightarrow$ filtering $\rightarrow$ training export}

\subsection{Preprocessing and Cleaning}

The repository includes data cleaning, audit, and filtering stages intended to make the temporal training set reproducible. These include:

\begin{itemize}[leftmargin=1.2cm]
  \item canonical temporal sample building,
  \item relation inventory generation,
  \item mined fact normalization,
  \item synthetic sample generation,
  \item quality filtering and distribution control,
  \item training-format export aligned with ChatKBQA instruction tuning.
\end{itemize}

\subsection{Train / Validation / Test Split}

TempQuestions provides the main held-out benchmark split, while the synthetic temporal dataset supplies training supervision. The split logic is intentionally separated into two layers. First, the \textbf{benchmark layer} uses the canonical TempQuestions train/test artifacts for reproducibility and for preserving a clean held-out human evaluation set. Second, the \textbf{model-selection layer} uses a lightweight holdout from the synthetic training pool for hyperparameter decisions before the final checkpoint is retrained on the full synthetic set.

\noindent Concretely, the canonical standardized TempQuestions artifacts contain \textbf{888 train samples} and \textbf{383 test samples}. The executed v1 model run did \emph{not} train directly on the corrupted full human train split. Instead, it used \textbf{2,061 synthetic temporal instruction-tuning examples} derived from 22 Zenodo relations. For tuning, this synthetic pool was treated as a \textbf{90/10 development split} (\textbf{1,855 train} / \textbf{206 validation}) so that beam size, epoch count, and learning-rate choices could be compared without touching the human test set. After selecting the final configuration, the locked checkpoint reported in this report was retrained on the \textbf{full 2,061-example synthetic set}. Final evaluation was then run on the exported human TempQuestions test artifact used by the temporal LLM workflow, which contains \textbf{268 test examples}. This split policy keeps model selection, final training, and final evaluation cleanly separated.

\subsection{Noise, Missing Data, and Bias}

Several data issues materially affect this project:

\begin{itemize}[leftmargin=1.2cm]
  \item corruption in the bundled TempQuestions processing path,
  \item incomplete or unusable temporal literals in some Freebase-aligned sources,
  \item bias toward explicit temporal expressions,
  \item archival staleness of Freebase,
  \item uneven entity and relation coverage.
\end{itemize}

The system addresses these through filtering, provenance logging, synthetic augmentation, and explicit reporting of limitations instead of silently claiming complete temporal coverage.

\section{Model Selection and Optimization}

\subsection{Chosen Model}

The project uses LLaMA-2-7B with LoRA fine-tuning for logical-form generation. This choice is pragmatic: it preserves the baseline family used by ChatKBQA, remains feasible on limited GPU hardware, and supports structured generation without retraining the entire base model.

\subsection{Why the Model Fits the Problem}

The model choice is justified by:

\begin{itemize}[leftmargin=1.2cm]
  \item strong instruction-following behavior for sequence generation,
  \item efficient adaptation via LoRA,
  \item compatibility with the existing ChatKBQA training scripts,
  \item easier checkpoint storage and deployment than full fine-tuning.
\end{itemize}

\subsection{Training Procedure}

The end-to-end training flow is:

\begin{enumerate}[leftmargin=1.4cm]
  \item prepare training examples in ChatKBQA instruction format,
  \item fine-tune LoRA adapters on the temporal training set,
  \item run beam search inference,
  \item post-process beam outputs,
  \item evaluate generated logical forms and downstream answer quality.
\end{enumerate}

\subsection{Optimization and Tuning Notes}

The repository documents practical tuning constraints such as environment compatibility, beam-search behavior, and operator imbalance. This project therefore reports a \textbf{basic engineering sweep} rather than claiming an exhaustive search. The sweep compared epoch counts of 3/5/10, learning rates of 3e-5/5e-5/1e-4, and beam sizes of 3/5/8 on the synthetic validation split. Three epochs underfit, ten epochs increased repetition, 1e-4 was less stable for logical-form syntax, and beam 8 increased runtime without a meaningful gain over beam 5. The selected configuration was therefore \textbf{5 epochs}, \textbf{5e-5}, and \textbf{5 beams}. These values are taken from actual run artifacts: the completed trial reached \texttt{max\_epoch = 5.0}, \texttt{max\_step = 645}, and 5-beam TempQuestions inference.

The answer-level score is low because the pipeline breaks after syntax generation. The model was trained on only 22 relations from Zenodo, but generated 177 unique relations at inference time. Most of these relations are hallucinated and do not exist in the evaluation knowledge base, so the generated S-expressions cannot be executed even when they look structurally valid. The ablation confirms the diagnosis in stages. First, a simple fuzzy grounding layer improves relation grounding from \textbf{0.5\%} to \textbf{88.9\%}, a roughly 178x increase, which proves that retrieval is essential. However, F1 still remains zero, which indicates a final limitation beyond entity linking: the relation semantics are often still wrong for the question, and the 22-relation Zenodo training slice does not cover enough of TempQuestions to support strong answer-level execution.

\subsection{Error Analysis}

The error analysis should focus on concrete system failure modes rather than generic model weaknesses. For this project, the most relevant categories are:

\begin{itemize}[leftmargin=1.2cm]
  \item incorrect logical-form structure,
  \item entity or relation retrieval mismatch,
  \item temporal operator misuse,
  \item incomplete date support in the data source,
  \item empty outputs that trigger fallback,
  \item benchmark-data quality issues.
\end{itemize}

\noindent In the current v1 run, the dominant category is \textbf{grounding failure}, with relation hallucination as the first visible symptom. The model was trained with only 22 relations, but generated 177 unique relations at inference time. Many of these look syntactically plausible while being absent from Freebase/Zenodo, such as \texttt{film.director.jedi\_master}, a long hallucinated \texttt{film.performance.*boyfriend} relation, and \texttt{foreign.exchange.rate.history.currency}. The three-way ablation then shows the deeper structure of the problem: relation grounding can be repaired substantially, entity grounding still fails without a linker, and even golden entities do not fully rescue execution because relation relevance and KB coverage remain limited.

Representative failures make the diagnosis concrete. For ``Where does Avril Lavigne live now?'', fuzzy grounding can map a relation into the KB, but the mapped relation points toward music tracks rather than residence, so the relation is executable but semantically wrong. For ``What was the first Harry Potter novel?'', the relation \texttt{/book/author/works\_written} is plausible, but the predicted entity MID points to the wrong author, so relation grounding succeeds while entity grounding fails. Generate-only outputs also include non-existent relations such as \texttt{film.director.jedi\_master} and \texttt{foreign.exchange.rate.history.currency}, confirming that retrieval-constrained grounding is required before this can become a reliable QA model.

\subsection{Trade-Off Discussion}

\textbf{Accuracy vs. speed:} larger beams and more retry logic may improve recovery but increase latency. \\
\textbf{Complexity vs. maintainability:} LoRA adapters and explicit routing are easier to maintain than a much larger monolithic redesign. \\
\textbf{Coverage vs. trustworthiness:} using more noisy synthetic or corrupted data may increase coverage but can reduce answer quality.

\section{Deployment Design}

\subsection{Supported Deployment Modes}

The repository already includes multiple deployment paths:

\begin{itemize}[leftmargin=1.2cm]

\subsection{Latency, Scalability, and Versioning}

Deployment quality should be discussed in software-engineering terms:

\begin{itemize}[leftmargin=1.2cm]
  \item expected latency on CPU and GPU,
  \item stateless API behavior and horizontal scaling implications,
  \item external service bottlenecks such as KB access and entity linking,
  \item model and config versioning through checkpoints and config files.
\end{itemize}

\noindent The submitted checkpoint is \texttt{models/LLaMA2-7b-tchatkbqa-trial/checkpoint}. For the verified 268-question, 5-beam inference run, the recorded inference time is approximately \textbf{12 minutes total}, which corresponds to about \textbf{2.3 seconds per question} on the remote GPU environment. This figure refers to inference throughput for the batch evaluation run rather than full API end-to-end latency under production serving conditions.

\subsection{Deployment Challenges and Limitations}

Important practical limitations include dependence on the knowledge base backend, entity-linking services, hardware requirements for inference, and the gap between local lightweight testing and remote heavy-compute execution.

\noindent The current codebase addresses this more explicitly than before by exposing a readiness distinction between \textbf{live-capable} and \textbf{degraded/offline} execution. In practice, a fully live run requires: (1) the base LLaMA model, (2) the LoRA adapter checkpoint, (3) Freebase/Virtuoso services, and (4) retrieval assets such as FACC1-derived entity files. When these are absent, the Streamlit demo now falls back transparently to recorded artifacts instead of pretending to be a fully live system. This is a more production-honest design because the user can see exactly what is missing.

Key drift risks include changing user language, new temporal patterns, benchmark mismatch, and knowledge-base staleness. \section{Privacy, Security, and Robustness}

\subsection{Privacy Considerations}

The project primarily uses public benchmark and knowledge-base data. However, once deployed, user queries themselves may become sensitive. The system should therefore minimize logging, avoid storing personal identifiers by default, and separate debug traces from any user-identifiable context.

\subsection{Security Considerations}

The main security discussion centers on API exposure, read-only KB access, environment secrets, and service hardening. The repository already avoids embedding obvious secrets, but deployment should still use environment-managed credentials and private network placement where appropriate.

\subsection{Robustness and Failure Cases}

The system should explicitly acknowledge failure on:

\begin{itemize}[leftmargin=1.2cm]
  \item out-of-domain questions,
  \item non-English inputs if not supported,
  \item vague temporal expressions,
  \item long or malformed prompts,
  \item questions about facts absent from Freebase.
\end{itemize}

The main mitigation is to make failures explicit: clear temporal questions should route to the temporal pipeline, non-temporal questions should preserve the baseline path, vague time wording should expose the selected operator, KB-missing facts should return provenance rather than overclaiming, and service dependency failures should be visible through health checks and bounded retries.

\section{Ethics and Responsible AI}

\subsection{Who Benefits and Who Could Be Harmed}

The system benefits users who need structured factual access, but it can also mislead users if they over-trust incomplete or outdated KB answers. Because Freebase is archived, the system can confidently return answers that are incomplete for post-2016 events.

\subsection{Bias and Fairness Risks}

Bias may appear through:

\begin{itemize}[leftmargin=1.2cm]
  \item uneven entity coverage,
  \item English-only focus,
  \item bias toward explicit temporal forms,
  \item dataset and retrieval artifacts inherited from legacy resources.
\end{itemize}

\subsection{Explainability}

One strength of this project is that the reasoning chain is more inspectable than in a pure black-box QA system. The answer can be accompanied by temporal signals, logical-form structure, SPARQL, and retry history. This does not make the LLM fully interpretable, but it does make the software decision path auditable.

\subsection{Potential Misuse}

Potential misuse includes over-reliance on outdated factual answers, uncritical reuse in high-stakes decision-making, and misuse of authoritative-looking outputs without verifying source limitations.

\section{Conclusion}

This project turns ChatKBQA into a temporal KBQA system with explicit attention to data quality, temporal reasoning, deployment readiness, and agentic control flow. The report positions the work as an engineering deliverable rather than a research-only prototype: code structure, training pipeline, deployment path, monitoring considerations, and responsible-use analysis are all part of the final artifact.

The final submission is therefore valuable in two ways. First, it implements a complete temporal KBQA stack with an auditable agentic routing layer. Second, it reports an honest benchmark diagnosis: syntax generation is already reasonably strong, while relation grounding, entity grounding, and KB coverage are the concrete bottlenecks that must be solved next.

\noindent In short, the v1 system is submission-ready as an engineering artifact, but not performance-ready as a production QA model. The correct next step is not to hide the 0.0 result, but to use the stage-wise evidence to justify retrieval-constrained generation, stronger entity linking, and broader Freebase relation coverage.
